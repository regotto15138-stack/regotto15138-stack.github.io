aof(文本保存指令, 存在丢失风险)
    执行指令 -> 指令存磁盘 // 不会阻塞当前操作
    写策略 
        always 影响性能
        everysec 丢失指令
        no 高性能 丢失指令
    指令合并(针对key-value多次操作, 记录key-value最新状态, 重写只用执行一次)
    主进程 fork 后台子进程 bgrewriteof 执行, 子进程 copy 主进程数据映射关系, 子进程执行aof(不会阻塞)

快照(原则上快照过程, 尽量避免修改数据, 若修改, 主进程copy一个副本, 在副本上修改, 子进程继续原有数据快照, 后续再重新更新至磁盘)
    save 主进程执行 阻塞
    bgsave 后台子进程
    快照使用增量快照, 初始进行全量快照, 后续增量操作
    redis4.0 混合aof+快照

redis数据同步
    主从模式, 主从读写分离
        主库写操作, 然后同步从库, 主从都可读操作
        从B上执行 replicaof A_ip 6379 B变成A的从库(redis5.0使用slaveof指令)
        主从全量同步过程:
            1.从库发送 psync ? -1 到主库, 主库回复 fullReSync 表示全量复制
                // redis实例启动会生成一个runId标识唯一性, 从库未知主库runId, 将runId标识?, -1代表offset, 表示第一次复制
            2.主库bgsave 生成rdb, 发送rdb到从库(发送过程主库可能写操作, replication buffer记录, 最后再让从库执行), 从库清空数据(避免脏数据), 然后加载rdb
        问题?
            1.从库较多, 导致主库fork子进程, 传输rdb影响性能
            答: 主->从1->从2模式, 主挂一个从1, 从1上挂从2, 减轻主库压力
                //库与库网络断开, 会导致主库从新全量复制, redis2.8后, 新增repl_backlog_buffer环形缓冲区, 主库记录写操作偏移量, 从库记录读操作偏移量, 若网络断开, 主库继续往环形缓冲区写, 当网络连接, 从库发送psync runId offset 主库copy增量数据到从库中.
                // 由于是环形缓冲区, 若主库写操作写满, 则会导致从库数据丢失, 主从间将重新开启全量复制, 在redis.conf配置repl_backlog_size大小, repl_backlog_size = (主库写入命令速度 * 操作大小 - 主从库间网络传输命令速度 * 操作大小)*2
                例: 如果主库每秒写入 2000 个操作，每个操作的大小为 2KB，网络每秒能传输 1000 个操作，那么，有 1000 个操作需要缓冲起来，这就至少需要 2MB 的缓冲空间。否则，新写的命令就会覆盖掉旧操作了。为了应对可能的突发压力，我们最终把 repl_backlog_size 设为 4MB
            2.主从复制为什么不选aof
            答:aof记录指令, 文件较大, 从库还需执行指令, 效率较低

哨兵机制(redis进程, 负责监控, 选主, 通知)
    监控:
        周期性ping主, 从, 判断是否下线
        ping操作后, 规定时间无响应, 哨兵将其标记为可能下线(存在网络问题导致误判, 引入多个哨兵, 少数服从多数, n个哨兵, 听从n/2+1个哨兵) 
    选主:
        按照一定规则从变主
        检查从与其他从的网络状态(检查断连阈值), 
        判断从库优先级(用户以及从库服务器配置, 设置slave-priority), 先选优先级高的
        从库与主库同步程度(增量同步中, 根据环形缓冲区主库写偏移量与从库偏移量, 先选读写偏移量最少的)
        根据runId, 默认情况, 优先级与复制进度相同, 先选runId最小的
    通知:
        选主后, 通知剩余从, 执行replicaof, 建立连接, 并全量复制, 通知客户端主库变更
    问题?
        1.主从切换, 客户端是否能正常操作
        答:读操作正常, 写操作失败, 失败时间=切换时间+客户端感知主库时间, 可将写的数据放入消息队列

redis6.0引出多线程
    多线程用于网络io解析(解析io数据), 解析后依旧是由主线程按照单线程的方式执行命令, 命令的执行结果再通过多线程形式写入socket缓冲区
    io-thread-do-reads yes//yes表示开启多线程
    io-threds 6//线程数, 该值要小于redis实例所在机器cpu数

redis使用规范
    1.key规范命名, 且长度不要过长
    2.避免出现bigKey, 将每个key-value的大小控制在10kb下, 对于hash, set集合元素, 将数据个数控制在1w下
    3.使用高效的序列化方案, 使用protostuff, kryo提前将数据序列化
    4.使用整数对象共享池(allkeys-lru, volatile-lru会统计每个key-value使用时间, 不同的key-value使用的是不同的, 不能整数对象共享)
    5.redis尽量保存热数据, 不同的数据保存到不同的实例中, 数据保存设置过期时间, 且添加过期范围时间, 避免集中过期
    6.线上禁用的指令:
        keys(使用scan代替, 返回符合条件的key-value), flushall(添加async后台异步操作), flushdb(添加async后台异步操作)会阻塞主线程
        monitor(导致服务端输出缓冲区溢出)
        避免进行全局操作, 使用sscan这种指令, 分批次返回数据, 减少阻塞
        使用mget, mset批量操作代替get/set
    7.redis部署的机器, 机器要保证性能, 主从情况下做读写分离, 实例设置最大连接数, 客户端连接使用连接池, 保持长连接, 扫描线上实例应设置操作时间, 避免影响业务

redis优化:
    根据业务场景, 自定义数据结构, 不同的业务线数据保存在不同的redis实例中, aof&rdb并存, redis冷数据缓存至数据库, 只保留热数据
