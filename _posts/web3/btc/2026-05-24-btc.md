## 密码学基础

| 技术 | 作用 |
|------|------|
| **哈希函数** | 签名验证，保证数据不可篡改 |
| **非对称加密** | 确立所有权 |
| **数字签名** | 防止双花（Double Spending） |

### SHA-256 哈希特性

- **不可逆**：无法通过哈希值反推出原文
- **抗碰撞**：很难找到两个不同的输入产生相同的哈希值
- **防篡改**：任何微小改动都会导致哈希值完全不同

### BTC 地址生成

```
BTC地址 = SHA-256(公钥) → RIPEMD-160
```

### 区块哈希链

```
curBlockHash = hash(preBlockHash + curData)
```

&gt; ⚠️ 修改 `preBlockHash` 会导致后续所有区块失效

## BTC 数据结构

### 区块（Block）

```yaml
block:
  blockHeader:
    version:              # 版本号
    previous_block_hash:  # 指向上一个区块的哈希
    merkle_root:          # 区块内交易的压缩指纹（轻节点只需验证一条路径，无需下载全部交易）
    timestamp:            # 时间戳
    bits:                 # 目标难度
    nonce:                # 随机数
  blockBody:              # 交易列表
```

### 交易（Transaction）

```yaml
transaction:
  input:
    previous_txid:   # 引用之前的交易
    output_index:    # 使用的是该交易的哪一笔输出
    scriptSig:       # 私钥签名（覆盖 input + output）
    sequence:        # 序列号
  output:
    value:           # 金额
    scriptPubKey:    # 锁定条件（定义谁能花费）
```

### UTXO（未花费输出）

&gt; UTXO（Unspent Transaction Output）是比特币的核心记账模型，只记录尚未被花费的交易输出。

## 协议机制

### 网络协议

- **P2P 广播**：去中心化网络传播

### 数据传播流程

1. 发送方广播 `block hash`
2. 接收方检查本地是否已存在
3. 若不存在，则请求并接受完整 block

### 交易验证规则

| 验证项 | 说明 |
|--------|------|
| 输入存在性 | 输入必须存在于 UTXO 集合中 |
| 未花费检查 | 每个 UTXO 只能花费一次 |
| 脚本验证 | `scriptSig` + `scriptPubKey` 执行结果为 `true` |
| 金额守恒 | 总输入金额 ≥ 总输出金额 |

### 区块验证规则

- [ ] 结构完整性检查
- [ ] `preBlockHash` 校验正确
- [ ] 满足工作量证明（PoW）
- [ ] `merkle_root` 未被篡改
- [ ] 逐条交易验证均合法
- [ ] Coinbase 奖励金额正确

### 共识机制

#### PoW（工作量证明）

```
hash(nonce) &lt; target
```

## 双花问题（Double Spending）

| 解决手段 | 说明 |
|----------|------|
| 全节点记录 | 所有节点保存完整的交易记录副本 |
| PoW 确认 | 工作量证明决定交易有效性 |
| 确认数 | 交易上链后，等待 **6 个区块确认** |

## 脚本系统（A → B 转账）

### 脚本类型

| 类型 | 作用 | 内容 |
|------|------|------|
| **解锁脚本** | 证明 A 可以花费该 UTXO | A 生成的签名 + A 的公钥 |
| **锁定脚本** | A 生成给 B 的新 UTXO，B 锁定该笔资金 | 定义花费条件 |

### 脚本验证示例

**找零给自己（A 的公钥哈希）**

```
OP_DUP OP_HASH160 &lt;A的公钥哈希&gt; OP_EQUALVERIFY OP_CHECKSIG
```

**转账给 B（B 的公钥哈希）**

```
OP_DUP OP_HASH160 &lt;B的公钥哈希&gt; OP_EQUALVERIFY OP_CHECKSIG
```

### B 后续使用 UTXO

&gt; B 需要根据交易生成新的签名，并提供公钥，证明之前 UTXO 中的资金属于自己。

```
&lt;B的签名&gt; &lt;B的公钥&gt; OP_DUP OP_HASH160 &lt;B的公钥哈希&gt; OP_EQUALVERIFY OP_CHECKSIG
```
